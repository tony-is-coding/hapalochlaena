## 2. 后端技术栈

### 2.1 运行时与语言

| 技术 | 版本 | 选型理由 |
|------|------|---------|
| **Node.js** | 20.x LTS | • 与 cc_core 同一运行时，进程内集成<br>• LTS 版本，稳定可靠<br>• 异步 I/O 性能优秀 |
| **TypeScript** | 5.x | • 类型安全，减少运行时错误<br>• 与 cc_core 技术栈一致<br>• 优秀的 IDE 支持 |

---

### 2.2 Web 框架

| 技术 | 版本 | 选型理由 | 替代方案对比 |
|------|------|---------|-------------|
| **Fastify** | 4.x | • **性能**：比 Express 快 2-3 倍<br>• **TypeScript 原生支持**：类型安全的路由和 schema 验证<br>• **插件生态**：JWT、WebSocket、CORS 等开箱即用<br>• **JSON Schema 验证**：内置请求/响应验证 | **Express**: 生态成熟但性能较差，TypeScript 支持不佳<br>**Koa**: 轻量但插件生态不如 Fastify<br>**NestJS**: 过于重量级，学习曲线陡峭 |

**Fastify 性能优势**:
```
Requests/sec:
- Fastify: ~76,000 req/s
- Express: ~30,000 req/s
- Koa:     ~50,000 req/s
```

---

### 2.3 数据库

| 技术 | 版本 | 选型理由 | 替代方案对比 |
|------|------|---------|-------------|
| **PostgreSQL** | 15.x | • **ACID 事务**：数据一致性保证<br>• **原子操作**：`UPDATE ... SET used = used + $1` 无需事务<br>• **JSON 支持**：灵活存储 permission_rules<br>• **成熟稳定**：企业级数据库 | **MySQL**: 功能相似但 JSON 支持不如 PostgreSQL<br>**MongoDB**: NoSQL 不适合强一致性需求<br>**SQLite**: 不支持并发写入 |

**为什么不用 ORM？**
- **性能**：原生 SQL 性能最优，无 ORM 开销
- **灵活性**：复杂查询（如 token 原子更新）更直观
- **学习曲线**：团队熟悉 SQL，无需学习 ORM DSL

---

### 2.4 数据库客户端

| 技术 | 版本 | 选型理由 | 替代方案对比 |
|------|------|---------|-------------|
| **node-postgres (pg)** | 8.x | • **原子操作**：直接执行 SQL，无 ORM 抽象<br>• **性能**：无 ORM 层开销<br>• **连接池**：内置连接池管理<br>• **简单直接**：SQL 即代码 | **Prisma**: ORM 抽象层，性能开销，原子操作不直观<br>**TypeORM**: 重量级，学习曲线陡峭<br>**Sequelize**: 老旧，TypeScript 支持不佳 |

**原子操作示例**:
```typescript
// node-postgres: 直接原子更新
await pool.query(
  'UPDATE token_ledgers SET used = used + $1 WHERE tenant_id = $2',
  [tokens, tenantId]
)

// Prisma: 需要额外逻辑
await prisma.$executeRaw`
  UPDATE token_ledgers SET used = used + ${tokens} WHERE tenant_id = ${tenantId}
`
```

## 7. 技术栈版本锁定策略

### 7.1 依赖版本管理

```json
{
  "dependencies": {
    "fastify": "^4.0.0",      // 主版本锁定，允许次版本更新
    "pg": "~8.11.0",          // 次版本锁定，只允许补丁更新
    "react": "18.2.0"         // 精确版本锁定（关键依赖）
  }
}
```

**策略**:
- **关键依赖**（React、TypeScript）：精确版本锁定
- **稳定依赖**（Fastify、PostgreSQL）：主版本锁定
- **工具依赖**（ESLint、Prettier）：允许次版本更新

---

### 7.2 升级策略

| 依赖类型 | 升级频率 | 测试要求 |
|---------|---------|---------|
| **安全补丁** | 立即 | 回归测试 |
| **次版本** | 每月 | 完整测试 |
| **主版本** | 每季度 | 完整测试 + 性能测试 |

## 8. 技术栈风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| **Fastify 生态不如 Express 成熟** | 中 | 使用官方插件，避免第三方插件 |
| **node-postgres 无 ORM 抽象** | 低 | 封装 Repository 层，统一数据访问 |
| **Vite 生态较新** | 低 | 使用稳定版本，避免实验性功能 |
| **Redis 单点故障** | 高 | 使用 Redis Sentinel 或 Cluster |
| **PostgreSQL 连接池耗尽** | 中 | 使用 pgBouncer 连接池管理 |
